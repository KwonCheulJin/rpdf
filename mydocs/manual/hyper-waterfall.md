# Hyper-Waterfall 방법론

## 개요

rpdf는 **Hyper-Waterfall** 방법론으로 개발됩니다. 이는 rhwp 프로젝트에서 차용한 접근으로, "거시적 워터폴 + 미시적 애자일"을 AI 페어 프로그래밍으로 결합한 것입니다.

## 왜 워터폴인가

전통적으로 워터폴은 느리고 경직된 방식이라 평가받습니다. 하지만 AI 페어 프로그래밍 시대에는 이야기가 달라집니다.

- **계획 비용이 극적으로 낮아짐**: AI가 계획서 초안을 빠르게 생성
- **구현 속도가 극적으로 빨라짐**: AI가 병렬로 다수 파일 수정 가능
- **결과**: 계획과 구현 사이의 비용 비율이 역전. 계획에 시간을 많이 써도 전체 속도는 빨라짐

## 왜 애자일도 필요한가

거시적으로 치밀하게 계획해도 구현 중에는 예상치 못한 문제가 반드시 생깁니다. 이 단계에서는 빠른 반복이 필요합니다.

## 이중 구조

```
┌─────────────────────────────────────────────────────────┐
│  거시 레이어 (워터폴)                                    │
│                                                         │
│  로드맵 → 마일스톤 → Issue → 계획서 → 구현 → 릴리즈     │
│                                                         │
│  - 아키텍처 결정은 이 레이어에서만                       │
│  - 방향 전환에는 명시적 승인 필요                        │
│  - 문서화 의무적                                         │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  미시 레이어 (애자일)                                    │
│                                                         │
│  계획서 → 코드 작성 → 테스트 → 피드백 → 코드 수정         │
│       ↑___________________________________|             │
│                                                         │
│  - Issue 범위 안에서만 이터레이션                        │
│  - 계획서 수정도 가능하지만 기록 남김                    │
│  - AI가 주도, 사람은 리뷰                                │
└─────────────────────────────────────────────────────────┘
```

## 실제 진행 예시

### 거시 레이어 타임라인

```
v0.1 (파서 뼈대)
  ├─ Issue #1 — 프로젝트 초기 세팅
  ├─ Issue #2 — PDF xref 파서
  ├─ Issue #3 — PDF 객체 파서
  ├─ Issue #4 — Content stream 파서
  └─ Issue #5 — IR 정의
v0.2 (렌더링)
  └─ ...
```

### 미시 레이어 (Issue #2 예시)

```
Day 1
  09:00 계획서 초안 작성 (AI)
  10:00 계획서 리뷰 (사람) → 수정 요청
  10:30 계획서 수정 (AI)
  11:00 승인 → 구현 시작
  15:00 xref 파서 코어 완성, 테스트 통과
  16:00 엣지 케이스 발견 (암호화 PDF)
  16:30 계획서에 엣지 케이스 추가 기록
  17:00 구현 완료
Day 2
  09:00 코드 리뷰 (사람)
  10:00 피드백 반영 (AI)
  11:00 PR → devel merge
```

## 각 단계별 산출물

### 1. 로드맵 (연 단위)
- 위치: `README.md`
- 포함: 버전별 목표, 방향성
- 수정 주기: 드물게, 전략적 결정 시

### 2. 마일스톤 (분기 단위)
- 위치: GitHub Milestones
- 명명: `M{버전}` (예: `M010` = v0.1.0)
- 포함: 마일스톤 내 타스크 목록

### 3. Issue (타스크 단위)
- 위치: GitHub Issues
- 포함: 문제 정의, 기대 결과, 제약 사항
- 마일스톤 지정 필수

### 4. 계획서 (Issue 단위)
- 위치: `mydocs/plans/task{N}-{slug}.md`
- 포함:
  - 목적 및 배경
  - 데이터 모델 변경
  - API 설계
  - 엣지 케이스
  - 테스트 전략
  - 예상 소요 시간
- 승인 없이는 구현 시작 금지

### 5. 오늘 할일 (일 단위)
- 위치: `mydocs/orders/yyyymmdd.md`
- 포함: 오늘 진행할 Issue 목록 및 우선순위

### 6. 완료 보고서 (Issue 단위)
- 위치: `mydocs/working/task{N}-done.md`
- 포함:
  - 실제 변경 내역 요약
  - 계획 대비 달라진 점
  - 배운 점
  - 다음 관련 작업

### 7. 트러블슈팅 (문제 단위)
- 위치: `mydocs/troubleshootings/{slug}.md`
- 포함: 문제 현상, 원인 분석, 해결책, 재발 방지

### 8. 기술 노트 (주제별)
- 위치: `mydocs/tech/{topic}.md`
- 포함: 기술 조사, 대안 비교, 의사결정 근거

## 불변 규칙

이 규칙은 프로세스 자체의 일관성을 위해 지켜야 합니다.

1. **계획서 없는 구현은 거부한다.** 계획서 작성을 건너뛰면 아키텍처가 오염됩니다.
2. **승인 없는 계획 변경은 거부한다.** 구현 중 계획을 벗어나고 싶으면 먼저 계획서를 수정하고 승인을 받습니다.
3. **아키텍처 결정은 반드시 사람이 한다.** AI는 선택지를 제시하고 장단점을 분석하지만, 최종 결정은 사람.
4. **완료 보고서 없는 머지는 없다.** 기록이 없는 변경은 나중에 재구성이 불가능합니다.

## 이 방식의 이점

| 일반 바이브 코딩 | 이 프로젝트 |
| --- | --- |
| 계획 없음 | 계획서 → 승인 → 실행 |
| 아키텍처 우연히 형성 | 의도적 설계 (CQRS, 의존성 방향) |
| 문서 없음 | 전체 과정 기록 |
| 디버깅 어려움 | 변경 이력 추적 가능 |
| 취약한 결과물 | 프로덕션 수준 품질 |

## 이 방식의 비용

솔직하게 말하면, 이 방식은 **혼자서는 느립니다**. 하지만 다음 상황에서 투자 대비 회수가 큽니다.

- 장기 유지보수가 필요한 프로젝트
- 시간이 지나 본인도 코드를 잊어버릴 위험이 있는 프로젝트
- 포트폴리오로 공개할 프로젝트 (과정 자체가 가치)
- 나중에 협업자를 받을 가능성이 있는 프로젝트

rpdf는 네 조건 모두에 해당합니다.

## 요약

> 거시적으로는 치밀한 계획, 미시적으로는 빠른 반복, 전체적으로는 투명한 기록.
