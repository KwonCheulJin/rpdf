# v0.4 — WASM 바인딩 및 웹 데모

**마일스톤**: M040
**목표 기간**: 3~4주
**상태**: 계획
**선행 조건**: v0.3 완료

## 목표

Rust 코어를 WebAssembly로 컴파일하여 브라우저에서 직접 사용 가능하게 만든다. `@rpdf/core` npm 패키지와 GitHub Pages 데모 사이트를 공개한다.

## 성공 기준

1. `wasm-pack build`로 pkg/ 디렉터리 생성 성공
2. npm 패키지 `@rpdf/core`로 배포 가능한 형태
3. 브라우저에서 PDF 파일 열기 → 썸네일 표시 → 병합 → 다운로드 동작
4. GitHub Pages에 데모 사이트 배포
5. WASM 번들 크기 2MB 이하 (gzip)

## 범위 외

- 데스크톱 앱 (v0.5)
- 완전한 에디터 UI (v0.6 이후)
- 오프라인 PWA (v0.7 이후)

### #26 PDFium의 WASM 대체 전략

**문제**: `pdfium-render`는 네이티브 PDFium 동적 라이브러리에 의존하므로 WASM에서 직접 사용 불가

**옵션 검토**
1. PDFium을 WASM으로 컴파일 (Google pdfium-wasm 프로젝트) — 복잡하지만 가능
2. 렌더링만 JS 측 pdf.js에 위임하고, Rust 코어는 파싱/편집만 — 간단하지만 이중 관리
3. 자체 SVG 렌더러 품질 끌어올림 — 시간 소모 큼

**선택**: 옵션 2. 웹 환경에서는 pdf.js로 렌더링, Rust 코어는 파싱/편집/저장만.

| 위험 | 가능성 | 영향 | 대응 |

## 기술 노트

### pdf.js vs Rust 파서 역할 분담

| 작업 | 담당 |
| --- | --- |
| 페이지 렌더링 (화면 표시) | pdf.js |
| 페이지 썸네일 | pdf.js |
| 문서 구조 파싱 | Rust 코어 |
| 편집 커맨드 | Rust 코어 |
| 저장 | Rust 코어 |

이 분담의 단점은 **같은 PDF를 두 번 파싱**한다는 점이다. v0.6 이후 pdf.js 의존을 줄이는 방향 재검토.

### Web Worker 활용

큰 PDF의 파싱·저장은 메인 스레드에서 하면 UI가 멈춘다. `@rpdf/core`의 호출을 Web Worker로 옮긴다.

```typescript
// core/wasm-worker.ts
import * as Comlink from 'comlink';
import initWasm from '@rpdf/core';

const api = {
    async open(bytes: Uint8Array) { ... },
    async rotate(handle: number, page: number, degrees: number) { ... },
    async save(handle: number): Promise<Uint8Array> { ... },
};

Comlink.expose(api);
```
