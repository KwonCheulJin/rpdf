# v0.3 — 편집 커맨드 (CQRS 레이어)

**마일스톤**: M030
**목표 기간**: 3~4주
**상태**: 계획
**선행 조건**: v0.2 완료

## 목표

`document_core` 레이어에 CQRS 패턴을 구축하고, 기본 편집 커맨드를 구현한다. 이 단계 이후부터는 PDF를 **변경 후 저장**할 수 있다.

## 성공 기준

1. `Command`, `Query` 트레이트가 정의되고 테스트 커버됨
2. `CommandStack`으로 Undo/Redo가 동작함
3. 최소 5개 커맨드 구현: merge, split, rotate, delete_pages, extract_pages
4. CLI로 모든 커맨드를 호출하고 결과 PDF를 저장할 수 있음
5. 저장 후 열었을 때 원본 품질이 보존됨 (pdfium 렌더링 결과 동일)

## 범위 외

- 주석 추가 (v0.6)
- 이미지 삽입 (v0.6)
- 기존 텍스트 편집 (v1.0 이후)

## 하위 Issue

### #15 Command/Query 트레이트 정의

```rust
pub trait Command: Send + Sync {
    fn execute(&self, doc: &mut Document) -> Result<CommandEffect>;
    fn undo(&self, doc: &mut Document, effect: &CommandEffect) -> Result<()>;
    fn name(&self) -> &'static str;
}

pub trait Query {
    type Output;
    fn execute(&self, doc: &Document) -> Result<Self::Output>;
}

pub struct CommandEffect {
    // Undo에 필요한 역정보
    pub undo_data: serde_json::Value,
}
```

### #16 CommandStack 구현

```rust
pub struct CommandStack {
    undo_stack: Vec<(Box<dyn Command>, CommandEffect)>,
    redo_stack: Vec<(Box<dyn Command>, CommandEffect)>,
    max_depth: usize,
}
```

- `execute`, `undo`, `redo` 메서드
- 최대 깊이 제한 (기본 100)
- 메모리 추적

### #17 RotatePageCommand
- 페이지 인덱스와 각도 입력
- 현재 `/Rotate` 값에 더해 정규화 (0/90/180/270)
- Undo: 반대 각도 적용

### #18 DeletePagesCommand
- 페이지 인덱스 리스트 입력
- 페이지 트리 재구성
- Undo: 삭제된 페이지 복원 (복원용 스냅샷 유지)

### #19 MergeCommand
- 여러 Document를 하나로 결합
- 각 페이지의 리소스 충돌 해결 (폰트, 이미지 중복 처리)
- Undo: 원래 페이지 수로 복귀

### #20 SplitCommand
- 페이지 범위 명세 (`1-3,5,7-10`)
- 결과로 다수의 Document 반환
- Undo: split은 원본을 변경하지 않으므로 no-op

### #21 ExtractPagesCommand
- Split의 간소화 버전
- 연속된 범위 하나만 추출하여 새 Document 반환

### #22 Serializer 구현
- `Document`를 PDF 바이트로 저장
- Incremental update 모드 (원본 뒤에 추가)
- Full rewrite 모드 (최적화, 압축)
- 옵션 선택 가능

### #23 CLI 편집 명령 추가

```
rpdf merge a.pdf b.pdf c.pdf -o merged.pdf
rpdf split input.pdf --pages 1-3,5,7-10 -o out_dir/
rpdf rotate input.pdf --page 3 --degrees 90 -o rotated.pdf
rpdf delete input.pdf --pages 2,4,6 -o deleted.pdf
rpdf extract input.pdf --pages 5-10 -o excerpt.pdf
```

### #24 편집 회귀 테스트

- 각 커맨드별 "입력 PDF → 커맨드 적용 → 결과 PDF"의 렌더링 스냅샷 비교
- Undo 후 원본과 동일한지 검증

## 데이터 모델 변경

v0.2까지는 `Document`가 read-only IR이었지만, v0.3부터는 `&mut Document`를 통해 변경 가능해진다. 변경 추적을 위해:

```rust
pub struct Document {
    // 기존 필드들
    pub pages: Vec<Page>,
    // ...

    // v0.3에서 추가
    pub(crate) dirty: bool,  // 변경되었는지
    pub(crate) original_bytes: Option<Arc<Vec<u8>>>,  // incremental save용
}
```

## 위험 요소

| 위험 | 가능성 | 영향 | 대응 |
| --- | --- | --- | --- |
| 병합 시 리소스 이름 충돌 | 높음 | 중 | prefix 재작성 로직 구현 |
| Undo 시 메모리 폭증 | 중 | 중 | 깊이 제한, 큰 변경은 페이지 단위 델타 저장 |
| `lopdf` 저장 후 열 때 뷰어 호환성 | 중 | 높음 | Adobe Reader, Preview, pdfium으로 교차 검증 |

## 테스트 전략

### 단위 테스트
- 각 Command의 execute/undo 대칭성 검증
- Query의 read-only 확인 (mutation 없음)

### 속성 기반 테스트
- 임의의 Command 시퀀스 + 동일 길이의 undo로 원본 복귀
- `execute → undo → execute → undo ...`의 결과가 일관됨

### 통합 테스트
- 실제 PDF로 end-to-end 워크플로우 (열기 → 편집 → 저장 → 다시 열기)
- 결과를 Adobe Acrobat Reader에서 수동 검증 (매 릴리즈 전 1회)
